## 4. 핵심 기술 결정 (Key Technical Decisions)

### 4.1 아키텍처 선택

| 결정 | 선택 | 근거 |
|------|------|------|
| 호스트 답글 모델링 | `Review.hostReply` 인라인 | 1:1 관계이므로 단순화, 별도 모델보다 효율적 |
| 이름 보호 | `maskName()` 함수 | 개인정보 보호 (GDPR 의식), 안심감 증대 |
| 리뷰 이미지 | 선택 필드 (String[]) | MVP에서는 UI 비노출, 추후 확장 가능 |
| SNS 크레딧 | Trx 처리 (리뷰+크레딧 동시) | 원자성 보장, 데이터 일관성 |
| API 설계 | `GET /api/reviews?propertyId=` | 범용 API로 재사용성 최대화 |

### 4.2 검증 전략

| 계층 | 검증 방식 | 구현 |
|------|---------|------|
| 클라이언트 | react-hook-form | ReviewForm/HostReplyForm |
| 서버 | Zod safeParse | `validations/review.ts` |
| DB | 제약 조건 | `bookingId @unique` |

### 4.3 에러 처리

```
400 Bad Request    → 유효성 검사 실패
401 Unauthorized   → 미인증 사용자
403 Forbidden      → 소유권 없음 (본인 예약 아님, 본인 숙소 아님)
404 Not Found      → 리뷰/예약/숙소 미존재
409 Conflict       → 이미 리뷰 작성함, 이미 답글 작성함
422 Unprocessable  → 체크아웃 미완료 예약
500 Server Error   → 트랜잭션 실패 등
```

## 6. 학습 사항 (Lessons Learned)

### 6.1 잘 진행된 사항 (What Went Well)

1. **PDCA 반복 효과**
   - 첫 번째 검증(v1.0)에서 72%로 부족했으나, Act 단계에서 즉시 6개 갭을 해결
   - 재검증(v2.0)에서 93% 달성 (21% 개선)
   - 반복 프로세스의 효율성 입증

2. **설계-구현 동기화**
   - Design 문서의 API 스펙, 컴포넌트 설계가 정확하여 편탈 구현 가능
   - 설계 문서와 구현 간의 gap이 최소화됨

3. **서버 검증의 중요성**
   - Zod 스키마를 명확히 정의하고 API에서 safeParse 사용 → 보안성 대폭 향상
   - 클라이언트 검증만으로는 불충분함을 재확인

4. **트랜잭션 처리**
   - 리뷰 작성 + SNS 크레딧 지급을 하나의 트랜잭션으로 처리
   - 원자성 보장으로 데이터 일관성 확보

5. **개인정보 보호 (Privacy)**
   - `maskName()` 함수로 게스트 이름을 간단히 마스킹
   - 신뢰 기반 플랫폼 구축에 기여

### 6.2 개선 기회 (Areas for Improvement)

1. **첫 번째 분석 정확도**
   - v1.0에서 72%로 평가했으나, 개선 후 93%까지 올라감
   - 초기 gap 식별 시 더 철저한 코드 리뷰 필요

2. **UI 텍스트 동기화**
   - Zod 스키마(500자) vs UI 텍스트(1000자) 불일치
   - 설계 단계에서 최대 길이 명시 필요

3. **페이지네이션 구현 완성도**
   - "더 보기" 버튼 텍스트만 있고 로직이 부분 구현
   - 설계 단계에서 구현 순서 명시 필요

4. **컴포넌트 분리 기준**
   - StarRating이 review-form 내부에 인라인 구현
   - 재사용성을 고려한 컴포넌트 추출 필요

### 6.3 다음에 적용할 사항 (To Apply Next Time)

1. **더 엄격한 설계 검증**
   - API 스펙, 컴포넌트 인터페이스, 에러 코드를 설계 단계에서 명시
   - 구현 전 설계 검토 체크리스트 활용

2. **단위 테스트 우선**
   - 각 API 엔드포인트별로 테스트 케이스 사전 작성
   - 구현 전 테스트 사례 정의 (TDD 원칙)

3. **코드 리뷰 루틴**
   - 구현 후 즉시 automated lint + type-check
   - 수동 코드 리뷰 체크리스트 운영

4. **UI/API 문서화**
   - API 요청/응답 예시를 Swagger/OpenAPI로 문서화
   - UI 텍스트 길이 제약을 설계 문서에 명시

5. **점진적 기능 릴리스**
   - 핵심 기능(P0) 완료 후 선택 기능(P1/P2) 릴리스
   - 사용자 피드백 반영 주기 단축
