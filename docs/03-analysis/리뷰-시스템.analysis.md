## 3. Gap Analysis (Design vs Implementation)

### 3.1 API Endpoints

| Design Endpoint | Implementation File | Status | Notes |
|----------------|---------------------|--------|-------|
| `POST /api/reviews` | `src/app/api/reviews/route.ts` (POST) | :white_check_mark: Match | 구현 완료, images/snsShareConsent 포함 |
| `GET /api/properties/[id]/reviews` | 미구현 (별도 라우트 없음) | :x: Missing | `GET /api/reviews?propertyId=...`로 대체 구현 |
| `POST /api/reviews/[id]/reply` | `src/app/api/reviews/[id]/reply/route.ts` | :white_check_mark: Match | 구현 완료 |
| `GET /api/host/reviews` | 미구현 (별도 API 라우트 없음) | :x: Missing | SSR 직접 Prisma 쿼리로 대체 |
| - | `GET /api/reviews` | :warning: Added | 설계에 없는 범용 GET 엔드포인트 추가됨 |

**상세 차이점**:

1. **GET /api/properties/[id]/reviews**: 설계서는 `GET /api/properties/[id]/reviews?page=1&limit=10` 전용 라우트를 명시했으나, 구현에서는 `/api/reviews?propertyId=...` 쿼리 파라미터 방식으로 대체. RESTful 리소스 경로가 아닌 쿼리 기반 필터링으로 변경됨.

2. **GET /api/host/reviews**: 설계서는 전용 API 라우트를 명시했으나, 구현에서는 `src/app/host/reviews/page.tsx`에서 Server Component로 직접 Prisma 쿼리 수행. API 라우트가 아닌 SSR 방식.

3. **응답 형식 차이**:
   - 설계: `{ reviews, pagination: { total, page, limit, hasMore }, summary: { avgRating, totalCount } }`
   - 구현: `{ reviews, averageRating: { average, count }, pagination: { page, limit, total, totalPages } }`
   - `hasMore` -> `totalPages`로 변경, `summary` -> `averageRating`으로 변경
