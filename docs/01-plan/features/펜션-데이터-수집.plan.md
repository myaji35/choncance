## 1. 배경 및 목적 (Background & Goal)

### 문제 정의
VINTEE 플랫폼은 촌캉스 숙소를 큐레이션하는 서비스이지만, 초기에는 호스트가 직접 등록한 숙소만 존재하여 **콜드 스타트 문제(Cold Start Problem)**가 발생한다. 숙소 수가 적으면 사용자 유입이 어렵고, 사용자가 없으면 호스트 유입도 어렵다.

### 해결 전략
외부 공개 데이터(공공 API, 웹 크롤링)를 통해 초기 숙소 데이터를 수집·가공하여 플랫폼의 콘텐츠를 선제적으로 채운다.

### 법적 안전장치 (CLAUDE.md §10.7 준수)
- 수집 데이터는 DB에 `source: "uploaded"`, `uploaded_by: "System Auto-Collection"` 으로 저장
- UI는 "파트너 숙소" 또는 "추천 숙소"로 표현
- 실제 크롤링은 백그라운드 Rails Rake Task로 처리
- 저작권 민감 콘텐츠(사진 등)는 원본 URL 참조만 저장, 직접 복사 금지

## 2. 수집 대상 및 우선순위 (Data Sources)

### P0 - 공공 API (즉시 시작, 합법)

| 소스 | API | 데이터 | 비고 |
|------|-----|--------|------|
| 한국관광공사 TourAPI | areaBasedList1 | 숙박 정보 (전국) | 무료, contentTypeId=32 |
| 공공데이터포털 | 농어촌 체험마을 현황 | 체험 프로그램, 위치 | 무료 CSV/JSON |
| 농어촌공사 그린투어 | 비공식 JSON | 농촌 관광 숙소 | robots.txt 확인 필요 |

### P1 - 웹 스크래핑 (검토 후 진행)

| 소스 | 방법 | 리스크 |
|------|------|--------|
| 한국 농촌체험마을 (welchon.com) | Nokogiri HTML 파싱 | robots.txt 허용 시 진행 |
| 강원도 관광 포털 | HTML 파싱 | 지역 공공 정보 |

### P2 - 향후 검토

- 여기어때 / 야놀자: robots.txt 비허용 가능성 높음 → 법적 리스크 있어 보류
- 에어비앤비: API 비공개 → 제외

## 3. 수집 데이터 스펙 (Data Schema Mapping)

수집 데이터를 VINTEE Property 모델로 매핑:

```
TourAPI 응답           →  VINTEE Property 필드
─────────────────────────────────────────────
title                 →  title
overview              →  description
addr1 + addr2         →  location, address
firstimage            →  thumbnail_url (URL 참조만 저장)
mapy, mapx            →  latitude, longitude
tel                   →  phone
-                     →  price_per_night: nil (미확인)
-                     →  status: "draft" (관리자 승인 필요)
-                     →  source: "uploaded"
-                     →  uploaded_by: "System Auto-Collection"
```

## 4. 기술 구현 계획 (Implementation Plan)

### Rails 아키텍처

```
app/
├── services/
│   ├── crawlers/
│   │   ├── tour_api_crawler.rb       # 한국관광공사 TourAPI
│   │   ├── public_data_crawler.rb    # 공공데이터포털
│   │   └── base_crawler.rb           # 공통 인터페이스
│   └── property_importer.rb          # 크롤링 결과 → DB 저장
lib/
└── tasks/
    └── crawl.rake                    # bin/rails crawl:properties
```

### 실행 흐름

```
bin/rails crawl:properties
  ↓
TourApiCrawler.new.call
  ↓
PropertyImporter.new(data).import!
  ↓
Property.create!(status: "draft", source: "uploaded")
  ↓
관리자 대시보드에서 검토 → status: "active" 승인
```

### 관리자 승인 플로우

```
/host/dashboard (관리자) → 미승인 숙소 목록 → [승인] / [거부]
```

## 5. 수용 기준 (Acceptance Criteria)

- [ ] TourAPI에서 농촌/펜션 숙소 최소 50개 수집 성공
- [ ] 수집된 데이터가 Property 모델에 정상 저장 (status: "draft")
- [ ] source: "uploaded", uploaded_by: "System Auto-Collection" 필드 확인
- [ ] 관리자가 draft → active 승인 가능
- [ ] Rake Task `bin/rails crawl:properties` 실행 성공
- [ ] 중복 수집 방지 (title + location 기준 upsert)
- [ ] 수집 실패 시 에러 로깅 (log/crawl.log)

## 6. 비기능 요건 (Non-functional Requirements)

- **Rate Limiting**: TourAPI 요청 간 1초 딜레이 (API 정책 준수)
- **배치 크기**: 1회 실행 시 최대 100건 (페이지네이션)
- **에러 복구**: 개별 항목 실패 시 skip 후 계속 진행
- **로그**: log/crawl.log에 수집 결과 기록
- **스케줄**: 초기 1회 수동 실행 → 이후 주 1회 cron 고려

---

## 7. 의존성 및 선행 조건 (Dependencies)

- **필수**: 한국관광공사 TourAPI 키 발급 (https://api.visitkorea.or.kr)
- **선행**: Property 모델에 source, uploaded_by 컬럼 마이그레이션
- **선행**: latitude, longitude 컬럼 마이그레이션 (현재 미존재 확인 필요)
- **라이브러리**: httparty gem (HTTP 클라이언트), nokogiri gem (HTML 파싱)
